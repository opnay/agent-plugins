# Design Brief

## Overview

Use this skill when the design problem is still vague and needs to become directionally clear before screens are specified.
Its job is to turn an idea, product request, or rough feature concept into a design-ready brief that a designer can work from without collapsing into implementation detail.
This skill defines what should be designed and why. It does not write code or produce detailed pre-code screen specifications by itself.

## Use When

- the idea is still fuzzy and needs design intent, hierarchy, and scope before screen planning
- the user wants a product or interface direction clarified in designer terms
- the main uncertainty is user outcome, content priority, tone, or non-goals
- the next step should eventually be a screen spec, but the directional brief is not locked yet

## Do Not Use When

- an existing design direction already exists and the real need is critique
- the directional brief is already aligned and the task is now detailed screen specification
- the user is asking for implementation guidance, component APIs, or code structure

## Core Policy

- Clarify the problem before proposing screens.
- Prioritize user outcome, hierarchy, and narrative clarity over premature layout detail.
- Name non-goals explicitly so the design direction has a clean stopping point.
- Make tone and content expectations explicit instead of leaving them implied.
- Keep the brief operational: it should be usable by a designer or planner as a next-step source of truth.

## Workflow

### 1. Define The Design Problem

Clarify:

- user or audience
- user outcome
- business or product goal
- core tension or design challenge

### 2. Lock Directional Boundaries

Make explicit:

- in-scope needs
- out-of-scope needs
- acceptable tradeoffs
- unacceptable tradeoffs
- constraints and assumptions

### 3. Shape The Design Direction

Specify:

- information hierarchy
- tone and interaction posture
- content emphasis
- what the design should feel clear, strong, or minimal about

### 4. Produce The Brief

Output a design-ready brief with:

- `Problem statement`
- `Audience`
- `Desired user outcome`
- `Design objective`
- `Information hierarchy`
- `Tone and UX posture`
- `Constraints`
- `Non-goals`
- `Open questions`
- `Recommended next step`

## Output Contract

- `Problem statement`
- `Audience`
- `Desired user outcome`
- `Design objective`
- `Information hierarchy`
- `Tone and UX posture`
- `Constraints`
- `Non-goals`
- `Open questions`
- `Recommended next step`

## Guardrails

- Do not jump into detailed screen structure before the direction is clear enough.
- Do not turn the brief into code requirements or engineering tasks.
- Do not substitute visual adjectives for actual design intent.
- Do not leave non-goals implicit.
