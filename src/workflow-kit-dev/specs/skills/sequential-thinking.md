# sequential-thinking 스킬 스펙

## 목적

`sequential-thinking`은 복잡한 분석, 설계, 계획, 디버깅 문제를 단계적으로 풀고, 필요하면 이전 판단을 수정하거나 대안 경로를 분기하며, 최종 결론 전에 가설을 검증하는 반성적 문제 해결 스킬입니다.
이 스킬은 외부 도구 호출이나 MCP 실행 환경을 요구하지 않고, 순차적 사고의 상태 전이 개념을 skill 절차와 작업 메모로 유지합니다.

## 경계

- 포함:
  - 순차적 문제 해결
  - 사고 단계 추정과 동적 조정
  - 이전 판단 수정과 대안 경로 분기
  - 가설 생성과 검증
  - 무관한 정보 걸러내기
- 제외:
  - workflow 선택용 라우터
  - 요구사항 발견 인터뷰
  - 읽기 전용 실행 계획 산출물 작성
  - 구현 자체
  - 숨은 사고 과정 원문 공개
  - MCP 도구 호출 또는 MCP server 설치 요구

## 처리하려는 작업 형태

- 복잡해서 한 번에 답하기 어려운 분석, 설계, 계획, 디버깅
- 중간에 방향 수정이나 대안 분기가 필요할 수 있는 문제
- 결론 전에 가설을 만들고 검증해야 하는 문제
- 범위가 처음부터 완전히 명확하지 않지만, 사용자 의도 발견보다 문제 해결 사고 절차가 병목인 경우

## 핵심 처리 계약

- 이 skill은 라우터가 아니라 문제 해결 흐름으로 동작한다.
- 현재 단계, 남은 단계 추정, 다음 사고 단계 필요 여부, 선택적 수정/분기/가설/검증 상태를 간결한 작업 메모로 유지한다.
- 단계 추정은 작업을 조정하기 위한 메모이며, 사용자에게 반드시 보여줄 출력 항목이 아니다.
- 특정 도구의 인자 형식을 사용자에게 요구하지 않는다.
- 숨은 사고 기록을 그대로 노출하지 않고, 사용자에게 필요한 판단 지점과 검증 결과만 보고한다.

## 흐름 기반 처리 계약

### 1. 진입 판단

- 문제의 병목이 workflow 선택이 아니라 순차적 문제 해결인지 먼저 판단한다.
- 사용자 의도, 범위, 절충점, 수용 기준이 병목이면 `deep-interview`로 넘기고, 필요한 가장 작은 질문 또는 인계를 명시한다.
- 실행 준비가 된 계획 산출물 작성이 병목이면 `planner`로 넘긴다.
- 단순 답변, 단순 구현, 최종 커밋 준비성 판단처럼 더 좁은 표면이 맞으면 이 skill로 확장하지 않는다.

### 2. 작업 메모 유지

- 현재 단계, 남은 단계 추정, 다음 사고 단계 필요 여부를 간결하게 유지한다.
- 수정된 가정이나 결론, 유지하거나 기각한 분기, 가설, 검증 상태를 작업 메모에 남긴다.
- 제약, 비목표, 증거의 한계, 소유권/절차 관련 증거, 자원 한계처럼 결론을 바꾸는 요소는 작업 메모와 결과 보고에 유지한다.
- 작업 메모는 내부 진행 상태를 정리하기 위한 것이며, 숨은 사고 과정 원문을 사용자에게 공개하기 위한 출력물이 아니다.

### 3. 순차 사고 진행

- 처음에 필요한 사고 단계 수를 추정하되, 실제 진행 중에 단계 수를 늘리거나 줄일 수 있다.
- 각 단계는 이전 단계의 결론을 그대로 반복하지 않고, 다음 판단에 필요한 새 정보, 가정, 검토 지점을 전진시킨다.
- 결론은 처음 추정한 사고 단계가 끝났을 때가 아니라, 충분한 답과 검증이 확보됐을 때 낸다.
- 무관한 정보는 흐름에서 제외하되, 제외 이유가 결론 이해에 중요하면 보고에 남긴다.

### 4. 수정과 분기

- 이전 판단이 틀렸거나 불충분하면 명시적으로 수정하고, 수정된 가정을 확정 사실처럼 쓰지 않는다.
- 의미 있는 대안 경로가 있으면 필요한 만큼 분기해 비교한다.
- 중요한 대안 경로를 기각하거나 보류한 경우에는 사용자가 결론을 이해하는 데 필요한 범위에서 그 이유를 보고한다.
- 분기는 문제 해결에 필요한 경우에만 사용하고, 가능한 경우의 수를 과도하게 늘리지 않는다.

### 5. 가설과 검증

- 결론 후보가 생기면 가설로 다루고, 가능한 검증 방법을 확인한다.
- 직접 검증이 가능하면 수행한 검증과 결과를 보고한다.
- 직접 검증이 불가능한 경우에는 완료된 검증처럼 쓰지 않고, 아직 제안 상태인 검증을 구분해서 보고한다.
- 더 넓은 제약이나 빠진 반례 확인이 중요한 판단에서는 좁은 단일 증거만으로 충분한 검증이라고 취급하지 않는다.

### 6. 제약과 다원인 분석 유지

- 권장이 코드 외 증거, 외부 제약, 소유권, 자원 한계에 의존하면 가장 편한 기술적 경로로 축소하지 않고 해당 제약을 권장에 반영한다.
- 여러 원인이 얽힌 분석에서는 절차, 소유권, 인계, 외부 증거가 중요하면 가장 쉬운 기술적 원인 하나로 축소하지 않는다.
- 비목표나 자원 한계가 결론의 형태를 바꾸면, 이를 단순한 부가 설명이 아니라 판단 조건으로 유지한다.

### 7. 보고와 인계

- 최종 보고에는 결론, 핵심 판단 지점, 수정 또는 분기, 검증 상태, 잔여 위험을 포함한다.
- 완료된 검증과 제안 상태의 검증을 구분한다.
- 다른 workflow로 넘겨야 하는 경우에는 넘기는 이유와 다음 표면을 명시한다.
- 사용자가 실행할 수 있는 결론을 제공하되, 숨은 사고 과정 원문을 그대로 노출하지 않는다.

- 실행 준비가 된 계획 산출물이 병목이라면 `planner`로 넘겼는가?

### 작업 메모와 순차 진행

- 이전 판단 수정, 대안 분기, 가설 검증 중 필요한 상태를 실제로 사용했는가?
- 숨은 사고 과정 원문을 노출하지 않고, 작업과 관련된 판단 지점만 보고했는가?
- 처음 추정한 단계 수에 억지로 맞추지 않고 필요한 만큼 조정했는가?

### 수정, 분기, 검증

- 완료된 검증과 제안 상태의 검증을 구분했는가?
- 기각한 중요 대안이나 무시/보류한 정보가 결론 이해에 필요하면 보고했는가?
- 결론 전에 필요한 검증을 수행했는가?

### 제약과 보고

- 비목표, 증거의 한계, 소유권/절차, 자원 한계가 결론에 영향을 주면 명시했는가?
- 더 넓은 제약이나 빠진 반례 확인이 중요한 판단에서 좁은 단일 증거를 과신하지 않았는가?
- 여러 원인이 얽힌 분석에서 절차, 소유권, 인계, 외부 증거를 가장 쉬운 기술적 원인으로 축소하지 않았는가?
