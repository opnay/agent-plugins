# workflow-kit-dev v0.1.1 Change Spec

## 변경사항 요약

- `autopilot` state guidance를 date-scoped `.agents/sessions/{YYYYMMDD}/` convention으로 정렬한다.
- `turn-gate`와 `turn-gate-self-drive`를 `workflow-kit-dev` 표면에서 제거하고, turn-level loop gate는 `loop-kit`의 operational package가 소유하도록 정리한다.
- `structured-thinking`을 제거하고, Sequential Thinking MCP server의 문제 해결 방식을 MCP tool 없이 동작하는 `sequential-thinking` skill로 변환한다.
- `sequential-thinking`의 revision, branching, dynamic step estimate, hypothesis verification 계약을 flow 기반 spec과 skill body로 고정한다.
- workflow skill을 직접 실행 표면으로 둔다.
- meaning resolution 대상에 `그 밑`, `그건`, `그거` 같은 한국어 지시 표현의 target ambiguity를 포함한다.

## 사용자 스펙 의도

- temporary or ad hoc session paths 대신 `.agents/sessions/{YYYYMMDD}/` convention을 사용해야 한다.
- `workflow-kit`에서 turn-level loop gate 표면을 제거하고, 그 책임은 `loop-kit`이 직접 소유하게 해야 한다.
- Sequential Thinking MCP server의 핵심 문제 해결 방식을 원본 출처와 함께 남기되, MCP tool wrapper가 아니라 독립 실행 가능한 skill 절차로 변환해야 한다.
- workflow router 역할로 흐르지 않고, 복잡한 분석/설계/계획/디버깅 자체가 병목일 때 쓰는 sequential, reflective problem-solving workflow로 유지해야 한다.
- 사용자의 `merge`, `병합`, `흡수`, `제거` 같은 표현이 skill surface, 절차 phase, routing rule 중 무엇을 가리키는지 모호하면 작업 전에 질문으로 잠가야 한다.
- manifest prompt, README, plugin spec에서 plugin 사용 기준을 분명히 하고 싶다.
- 사용자의 지시 표현이 주변 여러 section, 파일, skill, spec을 가리킬 수 있으면 target을 먼저 질문으로 잠가야 한다.
- 원본 출처, source URL, 사용자 스펙 의도 block은 대화 맥락처럼 버리지 않고 target ambiguity에서 보존해야 한다.

## 변경 목적

`v0.1.1`은 `workflow-kit`의 canonical workflow contracts를 직접 실행 가능한 workflow skill 중심으로 정렬하고, 불명확한 pre-workflow framing surface였던 `structured-thinking`을 명시적인 sequential problem-solving surface로 교체하는 patch입니다.
이 버전은 `autopilot` artifact location, workflow skill ownership, skill body composition을 같은 release line 안에서 정리합니다.
또한 Sequential Thinking MCP server의 출처를 유지하면서, MCP tool 호출 없이 동작하는 `sequential-thinking` skill/spec으로 문제 해결 절차를 승격합니다.
`loop-kit`은 이 canonical contract를 operational package로 흡수해 사용합니다.

## 변경 상세

### Autopilot Session Path

- `autopilot` state and QA issue guidance는 `.agents/sessions/{YYYYMMDD}/` 아래를 사용한다.
- 이 변경은 `advance-codex`의 `agents-sessions` folder convention과 충돌하지 않게 하기 위한 cross-plugin alignment다.

### Turn-Gate Surface Removal

- `workflow-kit-dev`에서 `turn-gate`와 `turn-gate-self-drive` skill/spec을 제거한다.
- turn-level loop gate와 self-drive overlay는 `loop-kit`의 operational package가 직접 소유한다.
- `workflow-kit-dev` manifest prompt, README, plugin spec에서 `turn-gate` 제공 문구를 제거한다.
- `workflow-kit-dev`은 `loop-kit`이 참조할 수 있는 일반 workflow skill 의미는 제공하지만, turn-level loop gate runtime surface는 제공하지 않는다.

### Structured Thinking Removal

- `structured-thinking` skill과 skill spec을 제거한다.
- workflow 선택이 가벼운 수준에서만 불안정한 경우는 manifest prompt와 plugin spec의 선택 기준이 직접 처리한다.
- requirement discovery가 병목이면 `deep-interview`가 질문 라운드를 소유하고, 단순 routing answer에서 종료하지 않는다.

### Sequential Thinking Skill

- `sequential-thinking` skill/spec을 추가한다.
- 원본 출처는 `https://github.com/modelcontextprotocol/servers/tree/main/src/sequentialthinking`이다.
- 이 skill은 MCP tool wrapper가 아니며, MCP tool 호출이나 MCP server 설치 없이 동작하는 self-contained skill conversion이다.
- 처리 흐름은 진입 판단, 작업 메모 유지, 순차 사고 진행, 수정과 분기, 가설과 검증, 제약과 다원인 분석 유지, 보고와 인계로 구성한다.
- `deep-interview`, `planner`, `commit-readiness-gate`, execution workflow와의 handoff 경계를 명시한다.
- 숨은 사고 과정 원문은 공개하지 않고, 사용자에게 필요한 판단 지점, 검증 상태, 잔여 위험만 보고한다.

### Meaning Resolution Checkpoint

- workflow 선택이나 파일 편집 전에 사용자 지시어의 operation 의미를 확인한다.
- `merge`, `absorb`, `remove`, `split`, `route`, `phase`, `surface`, `skill`, `spec`, `contract` 같은 표현이 여러 구조 단위를 가리킬 수 있으면 하나의 작업으로 단정하지 않는다.
- 해석 후보에 따라 수정 파일, 삭제 여부, phase 설계, routing rule, migration 의미, commit scope가 달라지면 user-gated question-routing으로 의미를 먼저 잠근다.
- workflow 선택 기준은 routing 전에 operation target ambiguity를 보존한다.
- 이 checkpoint는 requirement discovery가 아니라 현재 지시어의 의미 해독이며, `deep-interview`와 구분한다.
- 한국어 지시 표현이 여러 주변 대상을 가리키면 operation term이 명확해도 target ambiguity로 취급한다.
- source URL, provenance note, `사용자 스펙 의도` 또는 spec intent block도 meaning resolution에서 잠글 수 있는 작업 target으로 취급한다.
- 명시적인 commit 요청은 readiness, scope, staged/final status 확인 뒤 commit execution workflow로 handoff하고, readiness-only 요청은 commit 승인으로 취급하지 않는다.

### Plugin-Level Routing 정리

- workflow 선택 기준은 plugin spec, README, manifest prompt에 둔다.
- 각 workflow skill spec의 호출 방식 문구를 직접 호출 또는 manifest prompt 기준으로 갱신한다.
- `deep-interview-adaptation.md`는 plugin-level workflow 선택 기준과 결합되도록 갱신한다.

## 비목표

- `loop-kit`의 narrow operational packaging을 `workflow-kit` 안으로 흡수하지 않는다.
- turn-level loop gate를 `workflow-kit` 표면에 다시 추가하지 않는다.
- `autopilot`의 전체 loop semantics를 재설계하지 않는다.
- user-gated question routing을 제거하지 않는다.
- `sequential-thinking`을 workflow 선택용 router로 사용하지 않는다.
- Sequential Thinking MCP server를 설치하거나 호출하는 runtime integration을 추가하지 않는다.
- meaning resolution을 긴 requirement interview로 확장하지 않는다.
- 개별 workflow skill 자체를 제거하거나 합치지 않는다.

## 호환성 및 마이그레이션

- `workflow-kit-dev` manifest는 `0.1.0`에서 `0.1.1`로 올라간다.
- `autopilot` artifacts는 date-scoped `.agents/sessions/{YYYYMMDD}/` 아래에 둔다.
- `structured-thinking` 호출자는 더 이상 해당 skill을 사용할 수 없다.
- 복잡한 분석, 설계, 계획, 디버깅 자체의 reasoning이 병목이면 `sequential-thinking`을 사용한다.
- 요구사항 discovery가 병목이면 `deep-interview`를 사용하고, 실행 준비형 plan artifact가 병목이면 `planner`를 사용한다.
- `loop-kit`은 turn-level loop gate operational surface를 별도 소유한다.
- 사용자는 필요한 workflow skill을 직접 호출한다.

## 검증 기준

- dev/release `workflow-kit` manifests가 valid JSON이고 `0.1.1` version을 유지해야 한다.
- release `workflow-kit` surface에는 `specs/` directory가 없어야 한다.
- dev/release `autopilot` guidance가 `.agents/sessions/{YYYYMMDD}/` convention을 사용해야 한다.
- dev/release `workflow-kit` surface에는 `turn-gate`와 `turn-gate-self-drive` skill/spec이 없어야 한다.
- dev/release surface에 `structured-thinking` 참조가 남지 않아야 한다.
- dev/release `sequential-thinking` skill이 원본 MCP 출처, MCP-free conversion, revision/branching/hypothesis verification, handoff boundary를 설명해야 한다.
- `sequential-thinking` 12-scenario empirical rerun에서 critical checklist가 통과해야 한다.
- plugin-level workflow 선택 기준이 operation meaning ambiguity를 routing/작업 전에 질문으로 잠그는 계약을 설명해야 한다.
- dev/release surface는 manifest prompt에 적힌 workflow skill을 제공해야 한다.
- meaning resolution 24-scenario empirical run에서 critical checklist가 통과해야 한다.
- meaning resolution 40-scenario empirical run에서 발견된 partial case가 최소 보강 patch로 반영되어야 한다.

## 정식 규칙 승격 여부

- plugin/skill contract로 승격할 항목: `autopilot`, `sequential-thinking`, and other remaining workflow skill normative contracts are owned by their corresponding `specs/skills/*.md` files. Workflow selection guidance is owned by `plugin.md` and manifest prompt.
- repo-level rule로 승격할 항목: plugin-scoped skill 변경은 upstream canonical surface까지 함께 확인해야 한다는 원칙은 `AGENTS.md`가 소유한다.
- change spec에만 남길 이력: `0.1.1` 안에서 여러 follow-up commits가 같은 version surface를 갱신한 release history, and the removal/addition split used to keep `structured-thinking` removal separate from `sequential-thinking` addition.
