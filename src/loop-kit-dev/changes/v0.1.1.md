# loop-kit-dev v0.1.1 Change Spec

## 변경 목적

`v0.1.1`은 `loop-kit`을 `turn-gate` 중심의 좁은 turn-continuity plugin으로 정리합니다.
`turn-gate`는 기본 loop controller로 독립적으로 동작하고, self-drive는 준비된 sequence에 명시적으로 적용되는 별도 overlay reference로 둡니다.

## 사용자 스펙 의도

- `$loop-kit:turn-gate`를 사용한 turn은 사용자가 명시적으로 끝내기 전까지 `next-flow` phase를 열어야 한다.
- `turn-gate`의 기본 flow는 `preparation -> work -> verification -> reporting -> next-flow`이다.
- planned flow는 `분석`, `작업`, `검증`, `커밋 준비` 같은 phase checklist가 아니라, 함께 검토/검증/커밋 가능한 응집된 변경 단위여야 한다.
- 초기 preparation은 intent, scope, non-goal, acceptance signal, approval boundary, verification expectation을 충분히 수집하고 planned flow list를 만들 수 있어야 한다.
- 초기 의도 정렬과 planned flow list 설계 자체가 session plan, flow record, scope/approval boundary 산출물을 소유하는 `operational-preparation flow`가 될 수 있다.
- 실제 코드, 문서, fixture, 설정, release surface 변경은 `change-unit flow`로 분리한다.
- `deep-interview`, `review-loop`, `ralph-loop`, `autopilot`, `commit-readiness-gate`는 mode가 아니라 current phase에 적용하는 phase protocol이다.
- `turn-gate` 본체는 self-drive의 stop boundary, execution authority, handoff behavior, overlay 적용 조건을 소유하지 않는다.
- self-drive의 endpoint, stop boundary, execution authority, user-gated 복귀 조건은 spec-side `modes/self-drive.md`와 runtime `references/self-drive.md`가 각각 자기 표면에서 소유한다.
- self-drive가 명시적으로 적용되면 준비된 sequence의 진행 판단은 self-drive overlay 계약이 우선한다.
- verification은 작업 위험도에 맞춰 `clean-context`, `normal`, `not-required` method 중 하나를 선택해야 한다.
- verifier subagent의 사전 허용은 검증 전용이며, 파일 수정, scope 확장, destructive/external action, commit/push/PR/publish approval 생성에는 적용되지 않는다.
- clean-context verifier는 고위험 작업의 기본값으로 유지하되, 단순 질문 답변이나 work 중 이미 충분히 수행하고 기록한 command/check를 근거 없이 반복 실행하는 과한 검증은 줄인다.
- verifier packet은 flow 위험도, verification expectation, 이미 확보된 evidence에 맞는 최소 충분 범위로 구성한다.
- result reporting은 terminal close가 아니라 다음 flow 진행을 위한 continuity context다.
- visible question choices가 부족해도 turn-end option은 flow record에 남아야 한다.
- phase 시작 사용자-facing 메시지는 `[<phase-name>(/<phase-protocol>)]` 접두사로 시작하며, phase protocol segment는 optional이다.
- phase protocol segment의 optional은 protocol이 없을 때 생략한다는 뜻이며, phase protocol을 선택한 phase-start message에서는 suffix를 반드시 표기한다.
- 사용자 메시지 라우팅 layer를 제거하고, explicit stop은 terminal summary 허용 조건으로만 남긴다.
- `turn-gate`의 spec folder는 `core/`, `phases/`, `phase-protocols/`, `gates/`, `records/`, `modes/`, `templates/`, `intent-scenarios/`처럼 소유권 기준으로 정리한다.
- self-drive로 긴 planned flow sequence를 이어갈 때는 sequence objective, planned flow list, active flow index, allowed/prohibited autonomous actions, approval-sensitive checkpoints, endpoint, blocker return conditions, progress note를 기록해야 한다.

## 변경 범위

- `turn-gate` folder-based spec hierarchy를 유지하고, `spec.md`를 index로 둔다.
- `intent.md`는 사용자 스펙 의도만 소유한다.
- `core/runtime-flow.md`, `core/flow-boundaries.md`, `core/approval-boundary.md`, `core/meaning-resolution.md`, `core/skill-contents.md`가 loop 전체, flow 경계, 승인 경계, 의미 해석, runtime skill folder content boundary를 나눠 소유한다.
- `phases/*`는 phase별 세부 계약을 소유하고, `gates/*`는 내부 gate 전환 계약을 소유한다.
- `records/verification.md`, `records/question-routing.md`, `records/session-records.md`, `templates/*`가 verification packet, next-flow 질문, session record, template 계약을 나눠 소유한다.
- `phase-protocols/routes.md`는 implicit default operating state와 phase protocol routing만 소유한다.
- `modes/default.md`는 runtime-visible "default mode"가 아니라 기본 operating state 계약을 소유한다.
- `modes/self-drive.md`는 self-drive overlay 계약만 소유한다.
- runtime `SKILL.md`는 `turn-gate` 독립 loop controller로 재생성한다.
- runtime non-self-drive references는 phase protocol로만 작성한다.
- runtime `references/self-drive.md`는 self-drive overlay 계약으로 작성한다.
- runtime templates에서 self-drive-specific routing field를 제거한다.
- README, plugin spec, manifest prompt는 `turn-gate`와 self-drive overlay의 책임 경계를 반복 없이 설명한다.
- `turn-gate` spec 문서는 contract drift 없이 stale wording, phase 소유권 표현, self-drive 적용 조건 표현을 다듬는다.
- verification contract는 clean-context verifier 원칙을 고위험 작업의 기본값으로 유지하면서, 낮은 위험 작업에서는 normal evidence readback과 논리 반례 검토를 사용할 수 있도록 다듬는다.
- verification contract는 `clean-context`, `normal`, `not-required` method를 구분하되, result status는 `pass`, `fail`, `blocked`, `insufficient`로 유지한다.
- plan/flow record templates는 파일 경로에서 파생되는 date/path 값을 반복하지 않고, routing/state frontmatter와 Markdown 본문을 분리한다.
- flow record template frontmatter에는 stale closure 복구와 다음 행동 판단에 필요한 `continuity_note`를 포함한다.
- `000-plan.md`는 bounded date-level index와 active snapshot으로 유지하고, Flow Index와 Completed Flow Summaries는 flow당 한 줄 compact entry로 제한한다.
- `Planned Flow Sequence`에는 current/future selected flow만 남기고 completed flow detail은 compact index/summary와 `001+` flow record로 위임한다.
- runtime skill body와 spec map에서 이전 사용자 메시지 gate/section/spec 참조를 제거하고, stop handling은 별도 closure 조건으로만 설명한다.
- risk-based verification 평가를 위해 `micro` read-only, file-change verifier default, approval-sensitive release/version boundary, not-required routing intent scenario fixtures를 추가한다.
- runtime `SKILL.md`, README, plugin spec, manifest prompt를 three-level verification method 모델과 정렬한다.
- release `loop-kit` surface는 build command 산출물로 갱신한다.
- self-drive 전용 record shape는 일반 plan/flow template에 상시 노출하지 않고, runtime `references/self-drive.md`와 session-record guidance에서 조건부 기록으로 설명한다.
- self-drive sequence state는 optional 000-level sidecar인 `000-self-drive.md`가 소유하고, `000-plan.md`는 self-drive active 여부와 sidecar pointer만 소유하도록 정렬한다.
- self-drive sidecar ambiguity 보정은 `active_flow_index`를 0-based machine field로 명시하고, current flow label과 stale sidecar reconciliation 규칙을 추가한다.
- self-drive 실행 중 사용자 메시지가 들어온 경우의 interruption handling을 추가한다. 상태 질문은 현재 상태를 보고한 뒤 sequence를 계속하고, priority/scope/endpoint 변경은 updated sequence를 다시 잠그도록 준비 또는 next-flow routing으로 돌아가며, approval-sensitive 요청은 정확한 prior boundary 없이는 user-gated checkpoint로 보낸다.
- active self-drive 중 새 사용자 메시지는 `self-drive`를 다시 언급하지 않아도 self-drive mid-sequence input으로 처리하되, explicit stop과 approval/scope/endpoint guard는 계속 우선한다.

## 비목표

- `workflow-kit`의 broad workflow taxonomy를 `loop-kit` 안에서 재정의하지 않는다.
- direct loop entrypoints를 `loop-kit` user-facing surface로 늘리지 않는다.
- `turn-gate` 본체에 self-drive 세부 조건을 반복하지 않는다.
- release surface에 `specs/` 또는 `changes/`를 포함하지 않는다.
- installed Codex plugin cache refresh는 이 change spec의 release contract에 포함하지 않는다.

## 검증 기준

- dev/release plugin manifests가 valid JSON이고 version `0.1.1`을 유지한다.
- dev/release `turn-gate` skill이 `quick_validate.py`를 통과한다.
- release `loop-kit` surface에는 `specs/` directory가 없다.
- dev/release `skills/` surface가 build 후 일치한다.
- `modes/self-drive.md`를 제외한 `turn-gate` spec에는 self-drive/overlay 연결 문구가 남지 않는다.
- `references/self-drive.md`를 제외한 runtime skill folder에는 self-drive/self-driven/overlay/selected operating state/self-drive-handoff 문구가 남지 않는다.
- README, plugin spec, manifest prompt에는 `turn-gate`가 self-drive를 내부 mode로 소유한다는 표현이 남지 않는다.
- `git diff --check`가 통과한다.
- spec wording polish 이후 stale `work mode`, broad `analysis` bucket, reporting phase의 next-flow 질문 소유권 혼동이 남지 않는다.
- bounded plan policy 이후 `000-plan.md` template/spec/runtime이 detailed scope/evidence/verification을 반복하지 않고 compact index/snapshot 규칙을 일관되게 설명한다.
- routing demolition 이후 runtime `SKILL.md`와 specs에 이전 사용자 메시지 gate/section/spec 참조가 남지 않고, operation/target 해석과 approval/next-flow 판단은 preparation과 downstream 계약에 남는다.
- spec folder 정리 이후 `spec.md`와 child spec의 경로 참조가 새 `core/`, `phases/`, `gates/`, `records/` 구조와 일치한다.
- phase protocol suffix rule 이후 runtime `SKILL.md`와 specs가 optional segment 의미를 "protocol이 없을 때만 생략"으로 설명하고, protocol 선택 시 suffix 필수 표기를 명시한다.
- 검증 완화 이후 `records/verification.md`, `gates/verification.md`, `phases/verification.md`, plugin spec이 clean-context verifier 유지와 최소 충분 packet 원칙을 함께 설명한다.
- risk-based verification scenario fixtures가 no-edit read-only, file-change, approval-sensitive release/version, not-required routing 축을 구분하고, 각 축의 expected verification method와 forbidden behavior를 포함한다.
- three-level verification method 적용 이후 `not-required`가 result status나 automatic pass로 쓰이지 않고, reason/evidence/residual uncertainty 기록을 요구한다.
- runtime/plugin usage surface 정렬 이후 runtime body, README, plugin spec, manifest prompt가 `clean-context`, `normal`, `not-required`를 같은 의미로 설명한다.
- self-drive sequence record 적용 이후 runtime `references/self-drive.md`, session record reference, plan/flow templates, spec-side scenario가 sequence-level state와 flow-local snapshot의 소유권을 같은 의미로 설명한다.
- self-drive template exposure 보정 이후 runtime plan/flow templates에는 self-drive 전용 section이 없고, self-drive reference가 조건부 기록 위치와 필드 의미를 설명한다.
- self-drive sidecar 적용 이후 runtime에는 `templates/self-drive-template.md`가 포함되고, dev/release skill surface가 일치한다.
- self-drive ambiguity 보정 이후 `active_flow_index`의 기준, current flow label 병기, inactive plan에 남은 stale sidecar 처리, `000-plan.md`의 self-drive-specific pointer/status 소유권이 runtime/spec에서 같은 의미로 설명된다.
- self-drive mid-sequence routing 적용 이후 status-only user input, priority/scope change, approval-sensitive action request가 runtime/spec/scenario에서 같은 priority로 설명되고, 이 규칙이 broad user-message taxonomy가 아니라 self-drive interruption handling으로 남는다.
- implicit self-drive input 보정 이후 active self-drive 중 새 메시지가 `self-drive`를 반복하지 않아도 mid-sequence input priority를 적용한다고 설명하면서, 이 암묵성이 approval이나 scope 변경을 자동 승인하지 않는다고 분명히 한다.
- session record recovery 보정 이후 `not-yet-created` record와 active pointer가 가리키는 `unexpectedly missing`/`inaccessible` record를 구분하고, stale closure나 stale self-drive sidecar를 terminal close 또는 autonomous continuation authority로 사용하지 않는다고 spec/runtime/scenario가 같은 의미로 설명한다.
- non-pass verification routing 보정 이후 `fail`, `insufficient`, `blocked` status별 repair/blocker routing이 spec/runtime/scenario에서 직접 드러나고, 20-case fixture가 성공 보고/terminal summary/next-flow continuation 오용을 검출한다.
- read-only session record boundary 보정 이후 일반 target/source read-only와 workspace-wide no-write/no-record 제약을 분리하고, session record 운영 기록 허용/금지 조건을 spec/runtime/scenario가 같은 의미로 설명한다.
- post-commit scenario test 보정 이후 stale routing mismatch와 no-record read/write ambiguity를 runtime/spec/scenario에 명시해 fresh executor partial 판단을 제거한다.
- stale session record authority 보정 이후 inactive plan에 남은 sidecar, source-less/stale closure, future endpoint stop, plan/sidecar/flow mismatch, inaccessible/corrupt active records, stale approval boundary가 terminal closure 또는 autonomous continuation authority로 쓰이지 않도록 20-case fixture가 검출한다.
- not-required status/result boundary 보정 이후 `Method: not-required`가 automatic pass가 아니며 routing outcome, evidence, blocker, residual uncertainty에 따라 `pass`/`insufficient`/`blocked`가 분리된다는 점을 20-case fixture가 검출한다.
- runtime reference compression 보정 이후 runtime `SKILL.md`가 full dev scenario를 노출하지 않고도 phase-vs-flow, commit completion, self-drive status, future endpoint stop, file-change verification default, ambiguous operation trigger를 compact guidance로 직접 설명한다.
- conditional verification method fixture 보정 이후 조건부 method를 쓰는 scenario는 allowed methods와 selection criteria 또는 primary/upgrade trigger를 기록하고, strong fixture는 단일 expected method를 유지한다.
- explicit stop source matching 보정 이후 current explicit stop, future endpoint stop, status/report request, source-less/stale closure, compaction summary ambiguity를 20-case fixture가 분리해 검출한다.
- self-drive template machine readability 보정 이후 `000-self-drive.md` template frontmatter에 `current_flow_label`을 추가하고, `active_flow_index`와 사람이 읽는 current flow label을 함께 reconcile하도록 runtime/spec이 설명한다.
- self-drive progress note 보정 이후 `000-self-drive.md` template frontmatter에 current sequence summary인 `progress_note`를 추가하고, `Progress Ledger`는 append-only history로 분리한다.
- phase prefix application 보정 이후 phase-start/progress user-facing message와 self-drive automatic handoff에는 prefix를 요구하고, record/artifact/command output/question option 내부 prefix 오염은 20-case fixture가 검출한다.
- self-drive next-flow routing 보정 이후 active self-drive의 `next-flow` phase는 유효한 prepared sequence에서는 기록 기반 loop continuation이 되고, identity/scope/endpoint/approval/blocker ambiguity에서는 user-gated routing으로 돌아간다는 점을 runtime/spec/scenario가 같은 의미로 설명한다.
- question tool/autonomy boundary 보정 이후 self-drive는 질문 도구를 비활성화하지 않고 사용 조건을 좁히며, clear prepared transition 과잉 질문과 approval/scope/blocker 질문 부족을 20-case fixture가 함께 검출한다.
- unbounded self-drive endpoint 보정 이후 open-ended self-drive 요청도 finite cycle과 endpoint/repeat policy로 기록하고, finite exhaustion과 repeat cycle, unclear endpoint, explicit stop closure를 20-case fixture가 분리해 검출한다.
- dev/release path normalization 보정 이후 maintainer-facing README/spec은 `loop-kit-dev/...` shorthand가 plugin-root-relative이며 실제 편집 경로는 `src/loop-kit-dev/...`, 루트 `loop-kit/`은 build 산출 release surface라고 설명한다.
- session template path contract drift 보정 이후 `templates/*.md`는 dev spec-side contract owner이고 runtime 시작 파일은 `skills/turn-gate/templates/*-template.md`임을 `records/session-records.md`와 `templates/flow.md`가 설명한다.
- self-drive ownership naming 보정 이후 `modes/self-drive.md`의 `mode` 명칭은 dev spec-side taxonomy이고, 설치 후 실행 guidance는 runtime `skills/turn-gate/references/self-drive.md`가 소유한다고 plugin spec과 mode spec이 설명한다.
- flow parent plan pointer 보정 이후 flow record의 parent-plan relation은 같은 날짜 `000-plan.md`와 `Flow Index`에서 파생되며, 일반 runtime에서는 중복 `parent_plan` frontmatter를 요구하지 않는다고 runtime/spec이 설명한다.
- verification threshold for research artifacts 보정 이후 documentation-only research artifact 파일 변경은 clean-context 기본값을 유지하되, verifier packet은 changed topic/session file readback과 evidence gap 확인으로 좁힌다고 runtime/spec이 설명한다.
- flow record 원문 보존 보정 이후 `flow-record-template.md`의 `Flow Contract`는 사용자 메시지 원문과 요약/해석을 분리해 기록할 수 있고, 원문을 저장할 때는 재가공, 수정, 번역, 오탈자 정정, 완곡화, 맥락 병합을 하지 않는다고 spec/runtime template이 설명한다.
- Stop hook guard 승격 이후 `turn-gate`는 신뢰되고 활성화된 bundled Codex Stop hook을 optional runtime backstop으로 설명하되, hook이 conversation-level operating rule, reporting, record refresh, next-flow reopening 의무를 대체하지 않는다고 README/spec/runtime/manifest가 같은 의미로 설명한다.
- Stop hook guard 승격 이후 plugin hook feature 활성화, hook trust/reload, global Codex config는 자동 처리 범위가 아니며 user-gated setup으로 남는다고 설명한다.
- Stop hook spec ownership 보정 이후 plugin-level `specs/hooks/stop-hook.md`가 plugin runtime `hooks/hooks.json`와 `hooks/turn_gate_stop.py` 위치, manifest `hooks` field, quiet-exit 조건, block 출력, no-write 경계를 소유하고, `specs/skills/turn-gate/hooks/stop-hook.md`는 skill runtime guidance 작성 기준만 소유한다.
- Stop hook guidance dedup 보정 이후 `turn-gate` skill/spec/runtime guidance는 Stop hook의 detailed block/quiet-exit 조건을 반복하지 않고, optional backstop 의미와 block 후 main-agent 복귀 의무만 설명한다.
- Stop hook session arming 보정 이후 plugin-installed Stop hook은 payload의 current Codex session id와 `turn_gate_session_id` record가 일치할 때만 block할 수 있으며, session id가 없거나 다르면 fail-open으로 조용히 종료한다.
- SessionStart hook 추가 이후 plugin-level `specs/hooks/session-start.md`가 startup/resume context loader의 runtime file, discovery order, additionalContext output, no-write/no-approval 경계를 소유하고, `specs/skills/turn-gate/hooks/session-start.md`는 해당 context를 advisory startup context로 해석하는 기준만 소유한다.
- session record compacting 보정 이후 `000-plan.md`는 current recovery state, compact flow table, active open risks를 우선하고 긴 chronology, duplicate completed summaries, stale planned sequence, per-flow evidence는 `001+` flow record로 위임한다고 spec/runtime/template이 같은 의미로 설명한다.
- completed flow record compacting 보정 이후 empty/default fields, not-applicable risky action checklist, stale pending/next-flow state, verbose command checklist는 current recovery에 필요할 때만 유지하고 final snapshot 또는 historical detail로 낮춘다고 spec/runtime/template이 설명한다.
- skill folder regeneration contract 보정 이후 `core/skill-contents.md`는 `SKILL.md` section skeleton뿐 아니라 runtime `references/`와 `templates/`가 소유한 운영 계약을 runtime surface role, reference contract, template contract로 나눠 설명한다.
- skill folder regeneration contract 보정 이후 clean-context subagent가 스킬 폴더를 재작성하더라도 reference/template의 판단 기준, 소유권 경계, recovery state, approval limit을 짧은 요약으로 삭제하지 않아야 한다.
- regeneration drift 보정 이후 template specs와 `core/skill-contents.md`는 runtime template heading depth, self-drive `progress_note`/blocker return conditions, session-record recovery decision detail, self-drive question-tool/endpoint 판단 기준을 보존해야 한다고 명시한다.

## 정식 규칙 승격 여부

- plugin/skill contract로 승격할 항목: `turn-gate`는 turn continuity와 phase protocol routing을 독립적으로 소유한다.
- plugin/skill contract로 승격할 항목: self-drive는 별도 overlay reference로서 자기 stop boundary, execution authority, handoff behavior를 소유한다.
- repo-level rule로 승격할 항목: plugin-scoped skill 변경은 upstream/downstream related surfaces를 함께 점검해야 한다는 원칙은 `AGENTS.md`가 소유한다.
