# loop-kit-dev v0.1.1 Change Spec

## 변경사항 요약

- `turn-gate` flow record template and skill contract에 recorded turn-end option rule을 추가한다.
- `turn-gate-self-drive`를 별도 plugin skill overlay로 추가하고 base `turn-gate`에서 self-drive 세부사항을 분리한다.
- `turn-gate`를 conversation-level first-class control rule로 강화한다.
- `turn-gate` skill body를 spec-driven runtime surface로 재작성하고 release `loop-kit` surface를 동기화한다.
- `turn-gate` analysis에 사용자 지시어의 operation 의미를 먼저 해독하는 meaning resolution checkpoint를 추가한다.
- meaning resolution 대상에 `그 밑`, `그건`, `그거` 같은 한국어 지시 표현의 target ambiguity를 포함한다.
- 40-scenario evaluation에서 드러난 provenance, status interruption, commit handoff, self-drive approval boundary gap을 보강한다.
- activation-only bootstrap, explicit stop, correction reread, external action handoff 같은 turn lifecycle edge를 명시한다.
- `000-plan.md`를 current action checklist가 아니라 여러 flow의 계획과 전환 조건을 소유하는 date-scoped multi-flow plan으로 고정한다.
- `turn-gate-self-drive`가 flow-end question-routing 전에 `turn-gate`와 self-drive contract를 무조건 다시 읽도록 한다.
- `turn-gate`의 verification phase는 항상 clean-context subagent가 수행하도록 고정한다.
- 커진 `turn-gate` skill spec은 `specs/skills/turn-gate/spec.md`를 index로 두고, lifecycle, session records, meaning resolution, verification sub-spec으로 나눈다.

## 사용자 스펙 의도

- `$loop-kit:turn-gate`를 사용한 turn은 사용자가 명시적으로 끝내기 전까지 계속 next-flow gate를 열어야 한다.
- visible question choices가 부족해도 turn-end option은 flow record에 남아야 한다.
- self-drive는 `turn-gate`의 내부 문단이 아니라 같은 plugin 안의 별도 skill이어야 한다.
- `loop-kit` 변경은 upstream `workflow-kit` canonical contract를 확인하고 함께 맞춰야 한다.
- 사용자의 `merge`, `병합`, `흡수`, `제거` 같은 표현이 skill surface, 절차 phase, routing rule 중 무엇을 가리키는지 모호하면 internal mode 선택 전에 질문으로 잠가야 한다.
- 사용자의 지시 표현이 주변 여러 section, 파일, skill, spec을 가리킬 수 있으면 target을 먼저 질문으로 잠가야 한다.
- 원본 출처, source URL, 사용자 스펙 의도 block은 대화 맥락처럼 버리지 않고 target ambiguity에서 보존해야 한다.
- activation-only 요청은 결과 요약으로 닫히지 않고 다음 flow 선택으로 이어져야 한다.
- 사용자가 명시적으로 turn을 끝내면 terminal summary가 가능하지만, 그렇지 않은 correction, priority change, status check는 active flow를 계속해야 한다.
- `000-plan.md`는 phase별 작업 노트가 아니라 여러 개의 flow가 어떤 순서와 이유로 이어지는지 보여주는 plan이어야 한다.
- 예를 들어 "컴포넌트 오타가 있어. 수정하자"는 `컴포넌트 문구 점검`, `컴포넌트 문구 수정`, `commit-ready` 같은 여러 flow로 나뉘고, 각 flow는 자기 내부 단계와 다음 flow trigger를 가져야 한다.
- self-drive의 flow-end 질문 시점에서는 context cleanup이나 compaction 발생 여부를 감지했다고 단정하지 말고, 항상 `turn-gate`와 `turn-gate-self-drive` 계약을 다시 읽은 뒤 질문해야 한다.
- `turn-gate`의 검증 단계는 main agent가 같은 context로 직접 수행하는 단계가 아니며, 무조건 clean context 상태의 subagent가 수행해야 한다.
- main agent는 검증 요청을 구성하고 subagent 결과를 통합해 결과 보고, 보완 필요 여부, 다음 flow 판단을 결정해야 한다.
- `turn-gate` spec이 커진 경우 단일 파일에서 모든 세부 계약을 반복하지 말고 `specs/skills/turn-gate/spec.md`를 기본 index로 두고 세부 계약을 같은 folder 아래 sub-spec으로 분리한다.

## 변경 목적

`v0.1.1`은 `loop-kit`의 narrow operational package를 더 엄격한 turn-continuity controller로 정리하는 patch입니다.
`loop-kit`은 user-facing entrypoint를 `turn-gate` 중심으로 유지하면서, autonomous question routing은 `turn-gate-self-drive` overlay로 분리합니다.
이 release line은 `workflow-kit`의 canonical `turn-gate` contract를 흡수해 runtime skill body and templates에 반영합니다.

## 변경 상세

### Recorded Turn-End Option

- `turn-gate` skill/spec and flow record template에 recorded turn-end option rule을 추가한다.
- visible next-flow choices가 이미 가득 찬 경우에도 `Next Flow Options`에는 explicit turn-end option을 기록한다.
- 이 변경에서 `loop-kit-dev` and release `loop-kit` manifest version은 `0.1.1`로 올라간다.

### Self-Drive Overlay

- `turn-gate-self-drive` skill/spec을 추가한다.
- base `turn-gate` skill/spec/template에서 self-drive packet, answer, pause, recovery language를 제거한다.
- 기존 `turn-gate/references/self-drive.md` runtime reference를 제거하고 ownership을 overlay skill로 옮긴다.

### Plugin Skill Ownership Wording

- `turn-gate-self-drive`는 `turn-gate`의 표면이 아니라 같은 plugin 안의 별도 skill임을 명시한다.
- overlay는 같은 plugin의 `turn-gate` skill을 base continuity contract로 사용한다.
- 같은 wording을 `workflow-kit` canonical upstream에도 반영한다.

### Conversation-Level First-Class Rule

- `turn-gate`가 active turn의 conversation response itself를 제어하는 first-class control rule임을 명시한다.
- `user_explicit_stop`이 false인 동안 result reporting은 terminal summary가 아니며, next-flow gate를 다시 열어야 한다.

### Spec-Driven Skill Body

- `turn-gate` skill spec에 skill body composition contract를 추가한다.
- `turn-gate` skill body는 compact하지만 phase loop, internal mode selection, bad ending patterns, session record handling을 유지하는 runtime surface로 재작성한다.

### Meaning Resolution Checkpoint

- `turn-gate` analysis 단계는 internal mode 선택이나 작업 실행 전에 사용자 지시어의 operation 의미를 확인한다.
- `merge`, `absorb`, `remove`, `split`, `route`, `phase`, `surface`, `skill`, `spec`, `contract` 같은 표현이 여러 구조 단위를 가리킬 수 있으면 하나의 작업으로 단정하지 않는다.
- 해석 후보에 따라 파일 범위, phase 설계, routing rule, migration 의미, commit scope가 달라지면 user-gated question-routing으로 의미를 먼저 잠근다.
- 이 checkpoint는 `deep-interview`가 소유하는 requirement discovery가 아니라 current-flow instruction meaning resolution이다.
- 한국어 지시 표현이 여러 주변 대상을 가리키면 operation term이 명확해도 target ambiguity로 취급한다.
- source URL, provenance note, `사용자 스펙 의도` 또는 spec intent block도 meaning resolution에서 잠글 수 있는 작업 target으로 취급한다.
- status/progress check는 stop이나 새 독립 턴이 아니라 현재 phase와 progress를 보고한 뒤 active flow를 계속하는 입력으로 취급한다.
- 명시적인 commit 요청은 readiness, scope, staged/final status 확인 뒤 commit execution workflow로 handoff하고, readiness-only 요청은 commit 승인으로 취급하지 않는다.
- self-drive가 요청되면 autonomous question routing은 `turn-gate-self-drive`로 넘기고, approval, destructive, irreversible, external-action, safety 결정은 user-gated로 유지한다.

### Lifecycle Clarity

- activation-only bootstrap은 설정 확인만으로 닫지 않고, 다음 concrete flow를 여는 question-routing으로 이어진다.
- explicit stop은 사용자가 명시적으로 turn 종료를 요청한 경우에만 terminal summary를 허용한다.
- current-flow correction, priority change, status check는 stop이나 새 독립 턴이 아니라 active flow를 갱신하는 입력으로 취급한다.
- correction이나 priority change가 들어오면 필요한 local contract와 session continuity record를 다시 읽고 가장 이른 안전한 phase부터 재개한다.
- push, PR, destructive operation, external action은 user-gated approval boundary로 유지하고, commit handoff와 구분한다.

### Multi-Flow Plan Semantics

- `000-plan.md`는 날짜 단위의 여러 flow 계획을 소유한다.
- `000-plan.md`에는 flow sequence, 각 flow의 purpose, why, ownership, completion criteria, next-flow trigger, completed-flow summaries를 남긴다.
- 현재 phase의 세부 작업 증거와 pending checklist는 개별 `{count}-{slug}.md` flow record가 소유한다.
- `turn-gate` plan template은 `Planned Flow Sequence`, `Current Status`, `Completed Flow Summaries`, `Open Risks`를 통해 plan artifact와 flow record의 책임을 분리한다.
- typo-fix 같은 작은 사용자 요청도 필요하면 inspection, edit, commit-ready처럼 복수 flow로 나뉘며, `000-plan.md`는 그 전환 구조를 보여준다.

### Self-Drive Flow-End Context Refresh

- `turn-gate-self-drive`는 flow-end question-routing 전에 `turn-gate`와 `turn-gate-self-drive` instruction을 다시 읽는다.
- 이 refresh는 context cleanup이나 compaction을 감지했을 때만 실행하는 조건부 동작이 아니라, flow-end boundary마다 실행하는 deterministic step이다.
- refresh 대상 질문에는 next-flow choice, scope lock, self-drive packet 결정, user-gated approval boundary가 포함된다.
- reread 후에는 base continuity guard, self-drive routing limits, approval boundary를 반영해 질문을 구성한다.

### Clean-Context Verification

- `turn-gate`의 `work` 이후 verification phase는 항상 clean context 상태의 subagent가 수행한다.
- main agent는 같은 context에서 직접 검증을 완료했다고 단정하지 않고, 검증 대상, 사용자 의도, 변경 파일 또는 산출물, 실행할 command/check, pass/fail 기준을 subagent에게 전달한다.
- verification subagent는 구현이나 scope 확장을 맡지 않고 검증과 finding 보고만 수행한다.
- main agent는 subagent 결과를 통합해 residual uncertainty, 보완 필요성, 이후 flow/phase 재설계 필요 여부를 판정한다.
- subagent 검증이 실패하거나 불충분하면 result reporting 전에 active flow를 보완 단계로 되돌리거나 user-gated blocker를 연다.

### Nested Turn-Gate Spec Index

- `src/loop-kit-dev/specs/skills/turn-gate.md`는 `src/loop-kit-dev/specs/skills/turn-gate/spec.md`로 이동한다.
- `spec.md`는 `turn-gate` skill의 목적, 경계, 대표 표면, sub-spec map, internal mode selection overview, SSOT 동기화 원칙을 소유한다.
- 세부 계약은 같은 folder 아래 sub-spec으로 분리한다.
  - `lifecycle.md`: activation, response ending states, in-turn user intervention, explicit stop 처리
  - `session-records.md`: `000-plan.md`, `001+` flow records, Continuity Guard, templates, `Next Flow Options`
  - `meaning-resolution.md`: operation/target ambiguity와 user-gated clarification
  - `verification.md`: clean-context subagent verification, result reporting, next-flow reopening
- 이 구조는 direct loop mode skill을 늘리거나 broad workflow taxonomy를 `loop-kit` 안으로 가져오는 변경이 아니다.

## 비목표

- `workflow-kit`의 broad workflow taxonomy를 `loop-kit` 안에서 재정의하지 않는다.
- `turn-gate-self-drive`가 base `turn-gate`를 대체하지 않는다.
- direct loop entrypoints를 `loop-kit`의 user-facing surface로 확장하지 않는다.
- installed Codex plugin cache refresh를 이 change spec의 release contract로 포함하지 않는다.
- meaning resolution을 긴 requirement interview로 확장하지 않는다.
- `000-plan.md`를 단일 flow의 phase checklist나 command log로 사용하지 않는다.
- self-drive refresh를 compaction 감지 로직이나 hidden context-state detector로 구현하지 않는다.
- clean-context verification을 optional review, same-context self-check, 또는 commit-readiness 전용 절차로 축소하지 않는다.
- nested spec 구조를 release surface에 포함하지 않는다.

## 호환성 및 마이그레이션

- `loop-kit-dev` manifest는 `0.1.0`에서 `0.1.1`로 올라간다.
- self-drive behavior는 `turn-gate-self-drive` skill을 통해 진입한다.
- base `turn-gate` 호출자는 더 강한 continuity guardrail and recorded turn-end option behavior를 기대해야 한다.
- session artifacts now distinguish date-scoped multi-flow planning in `000-plan.md` from detailed per-flow records.
- self-drive callers should expect flow-end question routing to refresh base/self contracts before asking.
- `turn-gate` callers should expect every flow verification phase to dispatch a clean-context verification subagent before result reporting.
- release `loop-kit` surface is generated from `src/loop-kit-dev` and must stay spec-free.

## 검증 기준

- dev/release `loop-kit` manifests가 valid JSON이고 `0.1.1` version을 유지해야 한다.
- release `loop-kit` surface에는 `specs/` directory가 없어야 한다.
- dev/release `turn-gate` skill/template에 recorded turn-end option and conversation-level first-class guardrail이 있어야 한다.
- dev/release `turn-gate` skill/spec이 operation meaning ambiguity를 internal mode 선택 전에 질문으로 잠그는 계약을 설명해야 한다.
- meaning resolution 24-scenario empirical run에서 critical checklist가 통과해야 한다.
- meaning resolution 40-scenario empirical run에서 발견된 partial case가 최소 보강 patch로 반영되어야 한다.
- lifecycle clarity patch after scenario findings should pass a 20-scenario empirical rerun with no critical failures.
- `000-plan` semantics patch should pass targeted empirical scenarios for typo decomposition, priority insertion, and commit-ready transition.
- self-drive flow-end reread patch should pass targeted self-drive scenarios with no critical failures.
- dev/release `turn-gate` skill/spec must require clean-context subagent verification after `work` and before result reporting.
- `turn-gate` spec index must live at `src/loop-kit-dev/specs/skills/turn-gate/spec.md` and link to its lifecycle, session-records, meaning-resolution, and verification sub-specs.
- dev/release `turn-gate-self-drive` skill이 별도 plugin skill overlay ownership을 설명해야 한다.
- base `turn-gate` runtime references 아래에 removed self-drive reference가 남지 않아야 한다.
- release `loop-kit` surface must not include `changes/`.

## 정식 규칙 승격 여부

- plugin/skill contract로 승격할 항목: `turn-gate`, `turn-gate-self-drive`, and `loop-kit-dev-guide` responsibilities are owned by their corresponding specs and skill bodies. Meaning resolution is owned by `turn-gate`. Complex skill specs may use `specs/skills/<skill-name>/spec.md` as the index with child sub-specs.
- repo-level rule로 승격할 항목: plugin-scoped skill 변경은 upstream/downstream related surfaces를 함께 점검해야 한다는 원칙은 `AGENTS.md`가 소유한다.
- repo-level rule로 승격할 항목: nested skill spec index pattern은 `AGENTS.md`가 소유한다.
- change spec에만 남길 이력: `0.1.1` 안에서 turn-end recording, self-drive split, wording correction, first-class rule 강화, skill body rewrite, lifecycle clarity, multi-flow plan semantics, self-drive flow-end refresh, clean-context verification, nested spec index migration이 순차적으로 들어간 release history.
